# AIオーケストレーション パイプライン実装計画

## Context

Cognacのタスク自動化パイプラインは現在「ブートストラップ版」で、Phase 2（ペルソナ選定・ディスカッション・プラン策定）をスキップし、タスク作成後に直接Phase 3（コード実行）→ CI → completed と進む。

**現在の問題:**
- ブートストラップ版でもタスクが Pending → Stopped に直行する致命バグあり
- 原因: Claude CLIの仕様変更で `-p` + `--output-format stream-json` 使用時に `--verbose` フラグが必須になった
- エラー: `Error: When using --print, --output-format=stream-json requires --verbose`
- エラーログがダッシュボードUIで確認できない（console.errorのみ）

本計画では:
1. まず上記のバグ修正（`--verbose` フラグ追加 + エラーログのDB記録）
2. 設計ドキュメント（`doc/spec/DESIGN.md`）に基づき、Phase 2-A/B/C の完全実装
3. Git操作の接続
4. リトライコンテキストエスカレーション

を段階的に追加する。

**方針:**
- Claude CLI (`claude -p --output-format stream-json --verbose`) を維持。JSON抽出ヘルパー+リトライで信頼性確保
- `skipDiscussion` フラグで段階的にフェーズを有効化（既存のブートストラップモード維持可能）
- Git操作も今回のスコープに含める

### Step 0.5: エラーログのDB記録

**`server/runner/task-runner.ts`** (変更 — 小修正)

現状の `executeTask` の catch ブロック（111行目〜）で、エラーを console.error に出すだけで `execution_logs` に記録していない。以下を追加:

```typescript
// catch ブロック内
logQueries.createLog(this.db, {
  task_id: id,
  phase: 'execute',  // または発生したフェーズ
  error_type: 'app',
  error_message: err instanceof Error ? err.message : String(err),
})
```

これでダッシュボードのログタブからエラー内容を確認できるようになる。

### Step 5: Phase 2-A ペルソナ選定

**`server/runner/phase-persona.ts`** (新規)

```typescript
export async function executePhasePersona(
  task, db, config, onEvent?
): Promise<PhasePersonaResult>
```

- **プロンプト戦略**: 「開発チームリーダーとして2-4名の専門家を選出」+ JSONフォーマット指定
- **入力**: タスク情報 + repoStructure + taskHistory
- **出力**: `PersonaSelection` (personas[], estimatedRounds)
- **CLI引数**: `--max-turns 1`、`--dangerously-skip-permissions` なし
- **JSON抽出**: `extractJson<PersonaSelection>()` で抽出、失敗時は1回リトライ
- **フォールバック**: 最終的にJSON取れなければジェネリックペルソナ（コードレビューアー等）で続行
- **DB保存**: `personaQueries.createPersonas()` で保存
- **SSEイベント**: `persona_selected` を emit
- **実行ログ**: `logQueries.createLog()` で phase='persona' のログを記録

### Step 6: Phase 2-B マルチペルソナディスカッション

**`server/runner/phase-discussion.ts`** (新規)

```typescript
export async function executePhaseDiscussion(
  task, personas, db, config, onEvent?
): Promise<PhaseDiscussionResult>
```

- **ラウンドループ**: `round = 1` ~ `config.discussion.maxRounds`（デフォルト3）
  - ラウンド1: 各ペルソナの初期意見
  - ラウンド2+: 反論・補足・合意形成
  - `shouldContinue === false` で早期終了
  - 最終ラウンドでは強制的に `shouldContinue = false`
- **セッション継続**: ラウンド1でsessionId取得 → 以降 `--session-id` で引き継ぎ
- **プロンプト**: 全ペルソナをロールプレイさせる。各ラウンドの発言はJSONで返却
- **JSON抽出**: `extractJson<DiscussionRound>()` で抽出、失敗時はそのラウンドを1回リトライ
- **DB保存**: 各ラウンドで `discussionQueries.createDiscussionStatements()`
- **SSEイベント**: `discussion_round_start` → `discussion_statement` x N → `discussion_round_end`
- **実行ログ**: ラウンドごとに記録

export async function executePhasePlan(

### Step 12: task-runner.ts のパイプライン書き換え

**`server/runner/task-runner.ts`** (変更 — 最大の変更)

`executeTask()` を以下のフローに書き換え:

```
if (config.discussion.skipDiscussion) {
  // 既存ブートストラップフロー（Phase 3直行）
} else {
  // フルパイプライン
  Phase 2-A (discussing) → 2-B (discussing) → 2-C (planned)
  → Git branch作成 (executing)
  → Phase 3 + CI リトライループ
  → Git merge + push (completed)
}
```

**主要変更点:**
1. フルパイプラインの追加（`skipDiscussion` で分岐）
2. Git操作の接続（`createTaskBranch` → Phase 3 → CI → `mergeTaskBranch`）
3. リトライ時の `resetTaskBranch()` 呼び出し
4. リトライプロンプトエスカレーション（`buildRetryPrompt()` 使用）
5. タスク完了時の `invalidateContextCache()` 呼び出し
6. 各Phaseのエラーハンドリング（infra→paused、app→retry、process→retry）
7. `paused_phase` の正確な設定

**パイプラインの状態遷移:**
```
pending → discussing (2-A → 2-B → 2-C) → planned
→ executing (git branch → Phase 3) → testing (CI)
→ completed (git merge → push) | paused | stopped
```

## 注意事項・ベストプラクティス

### JSON出力の信頼性（CLIモード）

Web調査から得た知見:

1. **プロンプトで明確にJSON-onlyを指示** — 「JSONだけ返して、余計な説明はいらない」
2. **JSONスキーマ例を含める** — フォーマットの具体例を3-5個示すとClaude出力の安定性が大幅向上
3. **extractJson ヘルパーで多重フォールバック** — コードブロック → 生JSON → リトライ
4. **失敗時はフェーズ内で1回リトライ**（同じ入力で再実行）
5. **セルフチェック指示** — プロンプト末尾に「出力がJSONフォーマットに準拠しているか確認して」を追加

Sources:
- [Claude Structured Outputs](https://platform.claude.com/docs/en/build-with-claude/structured-outputs)
- [Claude Prompting Best Practices](https://platform.claude.com/docs/en/build-with-claude/prompt-engineering/claude-prompting-best-practices)

### オーケストレーション設計パターン

- **Sequential Pipeline**: Phase 2-A → 2-B → 2-C → 3 は順次実行。各フェーズが前のフェーズの出力に依存
- **専門化による信頼性向上**: 各ペルソナが専門領域に特化（セキュリティ、フロントエンド等）。汎用エージェントより専門エージェントの方が品質が高い
- **コードによるオーケストレーション**: LLMではなくコード（task-runner.ts）でフロー制御。決定論的で予測可能
- **エラー分類の3層構造**: app（リトライ可能）、infra（人間介入）、process（自動リトライ）

Sources:
- [AI Agent Orchestration Patterns - Microsoft](https://learn.microsoft.com/en-us/azure/architecture/ai-ml/guide/ai-agent-design-patterns)
- [LLM Orchestration Best Practices](https://orq.ai/blog/llm-orchestration)

## 検証方法

### 単体テスト
- `json-parser.ts`: 様々なClaude出力パターン（コードブロック、生JSON、混合テキスト）でJSON抽出が動作することを確認
- DB queries: 各テーブルのCRUD操作が正しく動作することを確認

### 統合テスト
1. `skipDiscussion: false` でタスクを作成し、パイプライン全体が動作することを確認:
   - pending → discussing → planned → executing → testing → completed の状態遷移
   - SSEイベントが正しく発火すること
   - DB にペルソナ・ディスカッション・プランが保存されること
   - Git ブランチが作成・マージされること
   - execution_logs に全フェーズの記録があること

2. `skipDiscussion: true` で既存のブートストラップモードが壊れていないことを確認

3. CI失敗時のリトライフローが正しく動作することを確認:
   - エラーコンテキストが escalate すること
   - ブランチリセットが行われること
   - 最大リトライ後に stopped になること

### 品質チェック
```bash
pnpm typecheck   # 全パッケージの型チェック
pnpm lint         # lint エラーがあれば修正
pnpm test         # テスト実行
```
