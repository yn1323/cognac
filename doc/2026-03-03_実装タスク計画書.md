# Solitary Coding - 実装タスク計画書

## Context

DESIGN.md に詳細に設計された「Solitary Coding」（AI駆動のタスク自動実行ツール）を、ゼロから実装する。現在リポジトリにはDESIGN.mdのみ存在し、コードは一切ない。

DESIGN.md のブートストラップ戦略に従い、まず最小限のパイプライン（タスク追加 → Phase 3実行 → CI → Gitマージ）を動作させる **ブートストラップフェーズ**（B-01〜B-14）を先行実装し、その後の機能拡充は **エンハンスメントフェーズ**（E-01〜E-15）として実装する。

ブートストラップの完了基準（DESIGN.mdより）:
1. `pnpx solitary-coding start` でサーバー起動
2. UIからタスク追加（title + descriptionのみ）
3. タスクランナーがキューから1つ取り出す
4. Phase 3（コード実行）が動く（Phase 2はスキップ）
5. CI実行（最低限lint + typecheck）
6. git branch → merge → push
7. エラー時の単純リトライ（5回）

### B-06: serverパッケージ - Claude CLI呼び出しヘルパー（callClaude）

DESIGN.mdの`callClaude()`共通ヘルパー関数を実装する。

**作成ファイル**:
- `packages/server/src/runner/claude-caller.ts`
  - プロンプトを`.solitary-coding/tmp/prompt-*.md`に書き出し
  - システムプロンプトを`.solitary-coding/tmp/system-*.md`に書き出し
  - `cat file | claude -p --output-format stream-json`でプロセス起動
  - `--append-system-prompt-file`（追加、上書きではない）
  - stream-json出力を1行ずつパースして`onStream`コールバックに渡す
  - stdoutタイムアウト（5分）でSIGTERM→SIGKILL
  - 完了後にtmpファイル削除
  - 共通トーンルール（日本語・カジュアル）をシステムプロンプトに自動マージ

**依存**: B-01, B-02

**検証**: tmpファイルの生成・削除、stream-jsonパース、タイムアウト時のSIGTERM送信

### B-11: serverパッケージ - タスクランナーオーケストレーター（Phase 3直接実行）

メインの処理エンジン。ブートストラップではPhase 2をスキップし直接Phase 3を実行する。

**作成ファイル**:
- `packages/server/src/runner/task-runner.ts` - TaskRunnerクラス:
  - start/stop/pause/resumeメソッド
  - キューポーリング: pendingタスクをqueue_order順に取得

  - リトライ: アプリ層最大5回、インフラ層でpaused、プロセス層最大2回

- `packages/server/src/runner/phase-execute.ts` - Phase 3ロジック:
  - タスク情報から実行プロンプト構築
  - `callClaude()` を `--dangerously-skip-permissions --max-turns 30` で呼び出し

**依存**: B-03, B-05, B-06, B-07, B-08, B-09, B-10

**検証**: pendingタスクの自動ピックアップ、Gitブランチ作成、Phase 3実行、CI成功時のマージ、5回失敗でstopped、インフラエラーでpaused

## 依存関係グラフ

```
B-01 (ワークスペース初期化)
 └─ B-02 (shared型定義)
     ├─ B-03 (DB層) ──┬── B-04 (REST API) ──┬── B-05 (SSE) ── B-14 (UI)
     │                │                      └── B-13 (start)
     │                └── B-11 (タスクランナー) ──── B-13
     ├─ B-06 (callClaude) ─── B-07 (streamパーサー)
     │    └── B-09 (CI)
     ├─ B-08 (Git ops) ────── B-11
     ├─ B-10 (エラー分類) ─── B-11
     └─ B-12 (init) ───────── B-13

[ブートストラップ完了]
 ├─ E-01 → E-02 → E-03 → E-04 (Phase 2統合、直列)
 │    ├─ E-05 (ディスカッションUI)
 │    ├─ E-06 (計画書UI)
 │    └─ E-11 (再実行機能)
 └─ E-07〜E-10, E-12〜E-15 (独立して並行可能)
```

## 推奨実装順序

**ブートストラップ**: B-01 → B-02 → [B-03, B-06, B-08, B-10, B-12 並行] → [B-04, B-07 並行] → B-05 → [B-09, B-14 並行] → B-11 → B-13

**エンハンスメント**: E-01 → E-02 → E-03 → E-04（直列）、E-05〜E-15は独立して並行可能

## 検証方法

1. **ブートストラップ完了の統合テスト**: テスト用リポジトリで `init` → `start` → UIからタスク追加 → Phase 3実行 → CI → マージの一連の流れが通ること
2. **各タスクの単体検証**: 各タスクの「検証」セクションに記載の基準
3. **型チェック**: `pnpm tsc --noEmit` が全パッケージで通ること
